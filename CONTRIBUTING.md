# Contributing

## Expectations

- Prefer clarity, correctness, and maintainability over cleverness.
- Work within the existing architecture and coding conventions unless there is a
  strong reason to change them.
- Keep changes small, explicit, and reviewable.
- Preserve type safety, deliberate error handling, and deterministic behavior.

## Development workflow

1. Create a branch from `main`.
2. Keep scope tied to one issue whenever practical.
3. Install development dependencies with `uv sync --dev`.
4. Run the full quality gate before committing:
   ```bash
   tox -e py312      # unit + integration tests
   tox -e lint       # ruff lint + format check
   tox -e type       # mypy strict type check
   tox -e pkg        # build sdist + wheel
   tox -e linecount  # 400-line per-file ceiling
   ```
5. Open a draft PR early if the work spans more than a small patch.

## Review standards

- Prefer straightforward logic over hidden magic.
- Avoid introducing optional abstraction layers before they are needed.
- Add or update tests when behavior changes.
- Document new user-facing behavior or workflow changes.
- **No source file in `src/reposage/` may exceed 400 lines.** When a file approaches
  the limit, split it by responsibility into a sibling module (e.g., `_dep_parsers_dotnet.py`
  splits the NuGet parsers from `_dep_parsers.py`). The `tox -e linecount` gate
  enforces this automatically.

## Commit and PR guidance

- Use descriptive branch names such as `feat/go-module-support`.
- Keep commits logically grouped — one logical change per commit.
- Summarize intent, validation, and follow-up risks in the PR body.
- No AI attribution lines (`Co-Authored-By:`, `Generated by`, etc.) in commit messages.

## Adding language support

See the **Extension Points** section of [docs/architecture.md](docs/architecture.md)
for a step-by-step recipe covering parser functions, manifest detection, framework
signals, quality signals, test fixtures, and tests. Follow the pattern established by
the Rust (M12) and C# (M13) implementations.

## First-time repository setup

Run the following once after cloning to create GitHub labels and milestones:

```bash
bash scripts/setup_github.sh
```

Requires the `gh` CLI to be authenticated.
